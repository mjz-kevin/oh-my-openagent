# oh-my-openagent 源码分析

> 一份对 **oh-my-openagent**（omo / oh-my-opencode，面向 OpenCode 与 Codex 的多 Agent 编排插件）源码的结构化解读文档。
>
> 从角色分工 → 目录结构 → 核心源码 → 架构分层 → 状态系统 → 完整工作流时序，逐层拆解。


## 一、Agent 角色介绍

### 1.1 三层协作模型

oh-my-openagent 由三个以希腊神话命名的 Agent 组成，形成「规划 → 编排 → 执行」的流水线：

```text
  Prometheus                    Atlas                          Sisyphus
  ┌─────────────┐              ┌─────────────┐               ┌─────────────┐
  │   规划者     │  ──计划──▶   │   编排器     │  ──任务──▶    │   执行者     │
  │   Planner    │              │ Orchestrator │              │  Executor    │
  └─────────────┘              └─────────────┘               └─────────────┘
```

### 1.2 Prometheus — 规划顾问

> **名字来源**：盗火给人类的泰坦，象征远见和智慧。

**职责**：

1. 接到一个模糊需求后，先探索代码库，搞清楚现有架构；
2. 只问那些探索解决不了的关键问题（不啰嗦）；
3. 研究最佳实践和设计模式；
4. 写一份决策完整的工作计划（`.omo/plans/` 下的 `.md` 文件）；
5. 计划写完后，等待用户批准。

> **一句话**：Prometheus 决定「**做什么**」。

### 1.3 Atlas — 主编排器

> **名字来源**：扛起天穹的泰坦，象征承载和协调。

**职责**：

1. 拿到 Prometheus 写好的计划；
2. 拆解成可并行执行的子任务；
3. 并行派出多个 Sisyphus 同时干活；
4. 验证每个 Sisyphus 的结果；
5. 循环直到所有任务完成；
6. 最后派出 Final Verification Wave 做最终审核。

> **一句话**：Atlas 决定「**谁来做、什么时候做、做完没**」。

### 1.4 Sisyphus — 任务执行者

> **名字来源**：推石头上山的国王，象征专注和坚持。

**职责**：

1. 接到一个具体、明确的任务；
2. 直接读文件、写代码、改代码；
3. 完成一个任务后汇报结果；
4. 如果失败了就重试（通过 `task_id` 延续上下文）。

> **一句话**：Sisyphus 负责「**动手做**」。

| Agent        | 角色定位   | 核心动作              | 决定的事                       |
| ------------ | ---------- | --------------------- | ------------------------------ |
| Prometheus   | 规划顾问   | 探索 → 写计划 → 等批准 | 做什么                         |
| Atlas        | 主编排器   | 拆任务 → 派发 → 验证  | 谁来做、什么时候做、做完没     |
| Sisyphus     | 任务执行者 | 读文件 → 写代码 → 汇报 | 动手做                         |

### 4.2 Hook 生命周期

每个 Hook 可以挂载到以下生命周期事件：

| 事件                    | 时机           | 典型用途           |
| ----------------------- | -------------- | ------------------ |
| `tool_execute_before`   | 工具执行前     | 拦截 / 修改        |
| `tool_execute_after`    | 工具执行后     | 验证 / 记录        |
| `message_before_send`   | 消息发送前     | 注入               |
| `message_after_receive` | 消息接收后     | 解析               |
| `session_start`         | 会话开始       | 初始化             |
| `session_end`           | 会话结束       | 收尾 / 续行触发    |
| `system_prompt_build`   | 构建系统提示时 | 注入角色指令       |

**关键 Hook 及挂载点**：

| Hook                       | 挂载事件                 | 作用                           |
| -------------------------- | ------------------------ | ------------------------------ |
| `prometheus-md-only`       | `tool_execute_before`    | 拦截 Write/Edit，只允许写 `.omo/*.md` |
| `keyword-detector`         | `message_before_send`    | 检测关键词，注入 skill 指令    |
| `atlas`                    | `system_prompt_build`    | 注入 Atlas 编排器指令          |
| `ralph-loop`               | `message_after_receive`  | 检测完成状态，自动续行         |
| `todo-continuation-enforcer` | `session_end`          | 检测未完成 todo，自动触发下一轮 |
| `start-work`               | `tool_execute_before`    | 读取计划，设置 Boulder 状态    |

### 4.3 Skill 加载机制

```text
用户输入 "/ulw-plan 做个博客系统"
  ↓
① keyword-detector hook 检测到 "ulw-plan"
  ↓
② 从 dist/skills/ulw-plan/SKILL.md 加载技能
  ↓
③ SKILL.md 被注入到 system prompt 中
  ↓
④ 我的行为被 SKILL.md 中的规则约束：
    - "你是 Prometheus，一个规划顾问"
    - "你只写 .omo/*.md 文件"
    - "探索代码库，判断意图"
    - ...
```

## 六、Boulder 状态系统

**作用**：持久化工作状态，跨会话 / 跨重启恢复。

interface BoulderState {
  schema_version: number;

  works: {
    [workId: string]: {
      work_id: string;
      active_plan: string;       // .omo/plans/<slug>.md

      elapsed_ms?: number;
      task_sessions?: Array<{
        label: string;
        title: string;
        elapsed_ms: number;
      }>;
    };
  };
}

## 七、完整工作流时序

下面以 `/ulw-plan 做个博客系统` 为例，展示从规划到执行完成的全流程：

```text
用户: "/ulw-plan 做个博客系统"
  │
  ├─→ keyword-detector 检测 "ulw-plan"
  ├─→ 加载 ulw-plan/SKILL.md  →  我变成 Prometheus
  ├─→ prometheus-md-only 激活  →  只能写 .omo/*.md
  │
  ├─→ 探索代码库
  ├─→ 判断意图（CLEAR / UNCLEAR）
  ├─→ 运行 scaffold-plan.mjs
  │     └─→ 生成 .omo/drafts/<slug>.md
  │     └─→ 生成 .omo/plans/<slug>.md（骨架）
  │
  └─→ 等待用户批准

用户: "批准"
  │
  ├─→ 追加 todo 到计划文件
  └─→ 提示 "请用 /start-work 开始执行"

用户: "/start-work"
  │
  ├─→ start-work Hook 激活
  ├─→ 读取 .omo/plans/<slug>.md
  ├─→ 写入 .omo/boulder.json（agent: "atlas"）
  ├─→ atlas Hook 激活  →  我变成 Atlas
  │
  ├─→ 读取计划，分析依赖
  ├─→ 并行派出多个 Sisyphus
  ├─→ 验证每个结果
  ├─→ 更新计划 checkbox
  │
  ├─→ todo-continuation-enforcer 检测未完成
  ├─→ 自动继续下一轮
  │
  └─→ 全部完成  →  ralph-loop 检测完成
       → 输出 ORCHESTRATION COMPLETE
```
